# 06. 安全 Runtime 设计

## 1. 核心原则

AI 不拥有最终写入权。AI 可以建议、规划、生成 patch，但最终写入必须经过 Runtime 校验和用户确认。

## 2. 修改边界

章节状态：

```text
draft：可大改。
unreviewed：可按规则优化。
reviewed：只能建议；应用需二次确认。
locked：禁止修改。
```

## 3. 文件权限

### AI 默认可读

- `book.spec.md`
- `outline.md`
- `style-guide.md`
- `rules.yaml`
- 当前章节 Markdown
- 当前相关批注
- `.bookai/chapter-status.yaml`

### AI 默认不可直接写

- `chapters/*.md`
- `rules.yaml`
- `.bookai/*.jsonl`
- `.git/*`

写入必须通过 Patch Engine / Rule Engine / Git Service。

## 4. Patch 校验

每个 PatchProposal 必须包含：

- 修改文件。
- 目标 block id。
- before_hash。
- after_text。
- 来源批注或明确用户指令。
- 修改原因。
- 使用规则。
- 影响范围。

如果缺少任何字段，进入 `invalid_patch`。

## 5. 自动拒绝条件

- 试图修改 locked 章节。
- 试图删除 block anchor。
- 试图覆盖整章但用户只授权段落。
- patch 无来源批注/用户指令。
- patch 改动 `.bookai` 元数据但不是 Runtime 工具调用。
- patch 改动 Git 文件。
- 批量传播规则时触达 reviewed/locked。

## 6. 二次确认条件

- 修改 reviewed 章节。
- 修改超过 N 个 block。
- 新增全局规则。
- 删除章节。
- 导出覆盖已有文件。
- 需要网络访问或外部模型调用。

## 7. Git 安全

每个接受的 patch：

1. 检查工作区是否干净。
2. 创建 pre-apply checkpoint。
3. 应用 patch。
4. 运行 block index 校验。
5. 写入操作日志。
6. 生成 Git commit。

commit message 示例：

```text
edit(ch05): apply AN-041 with R-018

- rewrite block ch05-p018
- source annotation: AN-041
- applied rule: R-018
- generated by skill: revise-with-annotations
```

## 8. 审计日志

`.bookai/audit-log.jsonl` 记录：

- run id。
- skill。
- model。
- 用户确认。
- patch id。
- commit hash。
- 被拒绝的修改。
- 安全策略拦截原因。
